# 包材收费技术方案

  

## 1. 背景与目标

  

当前寄付下单链路中，C++ `recipecache` 先计算药品重量，再调用 Go `express-service` 的 `ComputeShippingFee` 计算快递费。现有快递费只包含药品重量，不包含包材重量，也没有独立包材费。

  

本次目标：

  

- 包材费由我方系统独立收取，不进入顺丰计费。

- 包材重量并入药品重量，作为顺丰快递费的计费重量。

- 下单展示和收费项拆分为快递费、包材费。

- 下单、退款、对账关键链路输出包材计费详细日志，后续问题通过日志追溯。

  

第一阶段建议先完成“计算链路”闭环，退款和正式收费改造可独立排期。

  

## 2. 核心规则

  

### 2.1 计费重量

  

```text

药品重量 = C++ 现有 GetTotalAmont 逻辑计算

包材总重量 = sum(单个包材重量 * 包材数量)

顺丰计费重量 = 药品重量 + 包材总重量

```

  

### 2.2 包材费

  

```text

包材费 = 包材单价 * 包材数量

```

  

包材数量规则不建议在 Go 中硬编码。规则命中后，直接读取规则包材明细中配置的 `package_count`：

  

```text

包材费 = sum(明细.unit_price * 明细.package_count)

包材总重量 = sum(明细.package_weight * 明细.package_count)

```

  

例如 “15 剂以上使用 2 个包材” 不再依赖代码判断，而是配置两条剂数区间规则：

  

```text

1-15 剂规则：包材明细 package_count = 1

16-9999 剂规则：包材明细 package_count = 2

```

  

如果后续需要 “包装袋 1 个 + 冰袋 2 个”，同一条规则下配置多条包材明细即可。

  

### 2.3 规则命中维度

  

包材规则按以下维度命中：

  

```text

区域 + 处方类型 + 剂数匹配方式 + 生效时间

```

  

处方类型：

  

```text

1 = 代煎

2 = 自煎

3 = 代煎 + 自煎

4 = 膏方丸剂类

```

  

剂数匹配方式：

  

```text

1 = 剂数区间：按 dose_min <= dose_count <= dose_max

2 = 不限剂数：不判断 dose_min/dose_max

```

  

`recipe_type = 4` 表示膏方丸剂类，覆盖中药处方中的制膏、制丸、制散或协定方处方，以及颗粒剂处方等特殊包材配置类型。该类型配置时必须选择 `dose_match_mode = 2`（不限剂数），计算时不按剂数区间匹配。

  

## 3. 数据库设计

  

测试服务器 MySQL 版本为 `5.7.38-log`。该版本支持事务、普通索引、JOIN、IN、区间查询、`GET_LOCK`，不建议依赖 JSON 字段和 JSON 索引能力。

  

金额和重量字段均按项目现有风格使用整数存储：

  

- 金额使用 `BIGINT(20) UNSIGNED`。

- 重量使用 `BIGINT(20) UNSIGNED`，单位为克 `g`，不再额外放大。

- 现有 C++ `GetTotalAmont` 内部有 `*10` 中间计算，但返回前已 `/10`，传给 Go 的药品重量已经是实际克数。

- Go `ComputeShippingFee/GetShippingFee` 按克处理入参重量，`1000` 表示 `1kg`。

- 因此包材重量直接存克数，例如 `120g` 存储为 `120`。

  

包材配置表建议建在 `express-service` 当前快递配置所在数据库中。如果现有 `t_shipping_config` 位于 `db_external`，则以下表也建在 `db_external`：

  

```text

t_package_material

t_package_fee_rule

t_package_fee_rule_area

t_package_fee_rule_material

```

  

如果需要使用 autogen 生成代码，当前 SQL 主键字段是 `id`，生成命令的主键参数应使用 `-k id`；只有表字段改成 `Fid` 时才使用 `-k Fid`。

### 3.4 规则适用区域表

  

```sql

CREATE TABLE t_package_fee_rule_area (

    id BIGINT(20) UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',

    rule_id BIGINT(20) UNSIGNED NOT NULL COMMENT '包材收费规则ID',

    area_code VARCHAR(64) NOT NULL COMMENT '适用区域编码',

    area_name VARCHAR(100) NOT NULL COMMENT '适用区域名称',

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

  

    UNIQUE KEY uk_rule_area (rule_id, area_code),

    KEY idx_area_code (area_code),

    KEY idx_area_rule (area_code, rule_id)

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='包材收费规则适用区域表';

```

  

使用规则表 + 区域表的原因：

  

- 一条规则需要支持多个区域。

- 区域拆表后可用索引按 `area_code` 查询。

- 唯一性校验需要判断区域集合交集，拆表比 `gz|sz|bj` 或 JSON 字段可靠。

- 后续区域名称、状态、排序、层级扩展更容易。

  

### 3.4 订单最小记录与计费日志

  

订单侧不新增包材计费快照表，但需要在订单 `deal_extention` 中保存退款和金额追溯所需的最小包材计费值。完整命中规则、区域、剂数、重量计算过程通过下单、退款、对账关键链路日志追溯。

  

`deal_extention` 最小记录建议结构：

  

```json

[

  {

    "sf_material_code": "430300308905",

    "package_weight": 120,

    "package_count": 2,

    "unit_price": 300

  },

  {

    "sf_material_code": "430300308906",

    "package_weight": 50,

    "package_count": 1,

    "unit_price": 100

  }

]

```

  

字段含义：

  

- `sf_material_code`：本次使用的顺丰物料编码。

- `package_weight`：单个包材重量。

- `package_count`：本次使用数量。

- `unit_price`：单个包材收费金额，建议使用整数分。

  

根据这几个字段可以还原：

  

```text

单条包材费 = unit_price * package_count

单条包材总重量 = package_weight * package_count

整单包材费 = sum(单条包材费)

整单包材总重量 = sum(单条包材总重量)

```

  

日志建议结构：

  

```json

{

  "version": 1,

  "package_fee": 600,

  "drug_weight": 7200,

  "package_weight": 240,

  "billing_weight": 7440,

  "weight_unit": "g",

    {

      "rule_id": 10,

      "area_code": "GZ",

      "area_name": "广州",

      "recipe_type": 1,

      "dose_count": 18,

      "recipe_count": 2,

      "dose_match_mode": 1,

      "package_count": 2,

      "material_id": 1,

      "sf_material_code": "430300308905",

      "material_name": "纸箱|30cm_25cm_20cm|标准F3",

      "short_name": "F3纸箱",

      "unit_price": 300,

      "charge_unit": "个",

      "package_weight": 120,

      "total_package_weight": 240,

      "package_fee": 600

    }

  ]

}

```

  

这种方案的优点：

  

- 不新增订单侧快照表。

- 订单只保存退款需要的最小金额依据。

- 计算过程可通过日志追溯。

- 第一版改造范围更小。

  

需要注意：

  

- `deal_extention` 中保存的是最小计费值，不保存完整规则快照。

- 日志必须包含 `deal_id`、`rule_id`、区域、类型、剂数、包材数量、金额和重量。

- 如果合并寄付下一单包含多张处方，日志必须按同一包材计费分类汇总后输出明细。

- 后续如果运营需要按 `rule_id/material_code` 做大批量统计，再考虑补独立明细表或离线解析，不作为第一版强依赖。

  

## 4. 唯一性校验

  

唯一性不能依赖普通 `UNIQUE KEY`，因为冲突条件不是字段完全相等，而是多个维度存在交集：

  

```text

区域有交集

+ 处方类型有覆盖

+ 剂数匹配条件有交集

+ 生效时间有交集

= 冲突

```

  

### 4.1 区域冲突

  

新规则选择多个区域时，只要已有规则区域表中有任意一个相同 `area_code`，区域维度即冲突：

  

```sql

a.area_code IN (:new_area_codes)

```

  

### 4.2 类型冲突

  

`recipe_type = 3` 覆盖代煎和自煎；`recipe_type = 4` 为独立膏方丸剂类，不与 1/2/3 互相覆盖。冲突集合如下：

  

```text

新规则 recipe_type = 1，查询已有 1、3

新规则 recipe_type = 2，查询已有 2、3

新规则 recipe_type = 3，查询已有 1、2、3

新规则 recipe_type = 4，查询已有 4

```

  

技术方案中的“匹配时精确类型优先”与“唯一性禁止覆盖”存在语义冲突。第一版建议禁止覆盖共存，即只要类型覆盖就冲突，避免下单命中不稳定。

  

### 4.3 普通剂数区间冲突

  

普通剂数按闭区间判断交集：

  

```sql

r.dose_min <= :new_dose_max

AND r.dose_max >= :new_dose_min

```

  

示例：

  

```text

已有 1-7，新 7-10：冲突

已有 1-6，新 7-10：不冲突

已有 8-9999，新 7-10：冲突

```

  

### 4.4 不限剂数冲突

  

不限剂数规则不判断剂数区间。同一区域、同一类型覆盖、生效时间有交集时即冲突：

  

```sql

r.dose_match_mode = 2

```

  

`recipe_type = 4` 必须使用不限剂数，因此同一区域和时间内只能存在一条启用的膏方丸剂类规则。

  

普通剂数区间和不限剂数默认互相冲突，因为不限剂数覆盖全部剂数。

  

### 4.5 生效时间冲突

  

生效时间按闭区间判断交集：

  

```sql

r.effective_start_time <= :new_end_time

AND r.effective_end_time >= :new_start_time

```

  

即旧规则结束时间等于新规则开始时间时，也视为冲突。第一版建议使用闭区间，逻辑简单可靠。

  

### 4.6 冲突查询

  

```sql

SELECT r.id, r.rule_name, a.area_code, a.area_name

FROM t_package_fee_rule r

JOIN t_package_fee_rule_area a ON r.id = a.rule_id

WHERE r.status = 1

  AND r.id <> :current_rule_id

  AND a.area_code IN (:new_area_codes)

  AND r.recipe_type IN (:conflict_recipe_types)

  AND (

        r.dose_match_mode = 2

        OR :new_dose_match_mode = 2

        OR (

            r.dose_match_mode = 1

            AND r.dose_min <= :new_dose_max

            AND r.dose_max >= :new_dose_min

        )

      )

  AND r.effective_start_time <= :new_end_time

  AND r.effective_end_time >= :new_start_time

LIMIT 1;

```

  

`:new_area_codes` 和 `:conflict_recipe_types` 是动态数组，真实代码需要动态生成 placeholder，不能写死 3 个问号。

  

### 4.8 并发控制

  

保存规则时使用 MySQL 全局锁：

  

```sql

SELECT GET_LOCK('package_fee_rule_save_lock', 10);

-- 冲突查询

-- 保存规则主表

-- 保存区域表

SELECT RELEASE_LOCK('package_fee_rule_save_lock');

```

  

注意：`GET_LOCK` 是连接级锁。Go 实现时必须保证加锁、冲突查询、保存、释放锁使用同一个 MySQL 连接。不能 `db.Exec(GET_LOCK)` 后再 `db.Begin()`，因为连接池可能切换连接。

  

推荐伪流程：

  

```go

conn, err := db.Conn(ctx)

defer conn.Close()

  

locked := getLock(conn)

if !locked {

    return errors.New("当前有其他人正在保存包材规则，请稍后再试")

}

defer releaseLock(conn)

  

tx, err := conn.BeginTx(ctx, nil)

// conflict check

// insert/update rule

// insert/update rule areas

tx.Commit()

```

  

如果实际项目使用 gorm v1，需要确认同一连接能力；无法保证时，规则保存这段建议使用 `database/sql` 的 `Conn` 单独实现。

  

## 5. 下单计算链路

  

### 5.1 现有链路

  

当前 C++ `recipecache` 中，Go 计算快递费分支逻辑如下：

  

```text

CalExpressFee

  -> GetTotalAmont 计算药品重量

  -> GoCalExpressFee 调 Go ComputeShippingFee

  -> RenderCalExpressFee 返回 express_fee

```

  

现有 `GoCalExpressFee` 请求：

  

```text

{GO_URL}/express/handler/ComputeShippingFee

    ?clinic_id=...

    &dest_province=...

    &dest_city=...

    &weight=...

```

  

### 5.2 改造后链路

  

```text

CalExpressFee

  -> GetTotalAmontWithPackageInfo

       输出药品总重量，并按同一包材计费类型汇总剂数和药品重量

  -> GoCalPackageExpressFee

       输入区域、收件省市、汇总后的 calc_list

       Go 命中包材规则，读取规则包材明细，累加包材费和包材重量

       billing_weight = 药品重量 + 包材重量

       复用现有 ComputeShippingFee 逻辑计算快递费

  -> RenderCalExpressFee

       返回 express_fee、package_fee、billing_weight、package_detail

```

  

### 5.3 C++ 需要补充的数据

  

当前 `GetTotalAmont` 只返回总重量，不满足包材规则命中需要。建议新增结构，按同一包材计费类型汇总，不按单张处方逐条传给 Go：

  

```cpp

struct PackageCalcGroup {

    uint32_t recipeType;

    uint32_t doseCount;

    uint32_t recipeCount;

    uint32_t drugWeight;

};

  

struct PackageFeeCalcContext {

    uint32_t totalDrugWeight;

    std::vector<PackageCalcGroup> groups;

};

```

  

新增或扩展函数：

  

```cpp

uint32_t GetTotalAmontWithPackageInfo(

    const CCntlInfo& cntl,

    const ExpressInfo& expressInfo,

    PackageFeeCalcContext& calcContext);

```

  

不要在 C++ 里重复实现 Go 的包材规则命中逻辑。C++ 只做现有处方重量和必要处方信息提取。

  

汇总 key 建议为：

  

```text

recipe_type

```

  

示例：

  

```text

两张普通代煎处方分别 10 剂、8 剂
```

## 8. Deal 下单改造

  

现有 Deal 下单只接收 `ship_fee`，并写入 `FEE_TYPE_SHIP`。

  

后续正式收费需要：

  

- 请求参数新增 `package_fee`。

- 订单总价纳入包材费。

- 费用明细新增包材费 fee type，例如 `FEE_TYPE_PACKAGE_MATERIAL`。

- `CheckShipFee` 或新增校验逻辑同时校验快递费和包材费。

- `SplitRecipeAdd` 医保/自费拆单时明确包材费归属。

- 下单成功后写入包材最小计费记录，并输出包材计费详细日志。

  

不要把包材费混进 `ship_fee`，否则退款、对账、展示都会和快递代收费混淆。

  

## 9. 退款设计边界

退款规则：

```text
未发货整单退：包材费全退
未发货部分退：包材费不退，按整单计
已发货退款：包材费不退
合并寄付：包材费按订单计费明细退，不能简单按单张处方均摊
```

现有 Deal 中：

- `PartRefundCheck` 是部分退款预计算。
- `CancelAndWXRefund` 是实际退款执行。
- 部分退款结果会写 Redis，实际退款时按预计算结果执行。

**整单退（已发货不退包材费）实施方案见 §15.6。**

部分退款包材逻辑（未发货不退、合并寄付分摊等）属于后续迭代，暂不排期。如需实现，则要进入 `PartRefundCheck` 的预计算结构、`CRecipeRefundInfo`、Redis 序列化，范围更大。

  

## 10. 风险与待确认项

  

### 10.1 剂数边界

  

剂数边界不再由 Go 硬编码，而是通过规则表配置。例如产品确认 “15 剂以上” 的口径后，直接配置对应剂数区间：

  

```text

如果 15 剂本身算 1 个包材：1-15、16-9999

如果 15 剂本身算 2 个包材：1-14、15-9999

```

  

Go 只按命中的规则读取 `t_package_fee_rule_material.package_count`。

  

### 10.2 区域来源

  

包材规则的区域必须明确是：

  

```text

门店/发货区域

还是收件区域

```

  

建议使用门店/发货区域。

  

### 10.3 金额单位

  

包材单价、包材费等金额字段使用 `BIGINT(20) UNSIGNED`，和订单表金额字段保持一致。Go 侧不要使用 `float64` 参与金额计算，auto_gen 后需要确认 `UnitPrice` 生成的是整型。

  

### 10.4 重量单位

  

包材重量字段使用整数克数存储，不使用 `DECIMAL`，也不做额外倍率放大。需要保证：

  

- 后台录入和展示均按克 `g` 处理。

- Go 规则计算直接使用 `package_weight * package_count` 得到包材总克数。

- C++ 传入的药品重量是克，`drug_weight + total_package_weight` 后可直接传给 Go 快递费计算。

- 日志中明确输出药品重量、单个包材重量、包材数量、包材总重量、最终计费重量，单位均为克。

  

### 10.5 试算与下单一致性

  

试算和下单之间规则可能变化。可靠方案：

  

- 试算返回命中规则和金额。

- 下单时重新计算或校验前端传入金额。

- 下单成功后写入包材最小计费记录，并输出完整计费日志。

  

如果只依赖前端传回的 `package_fee`，存在被篡改或规则变化导致不一致的风险。

  

### 10.6 Go 伪代码落地差异

  

现有 `express-service` 使用 pb handler 和 gorm 风格，技术文档里的 REST 路径和 `database/sql` 代码只是伪代码。真实落地时需要按项目现有接口体系实现。

  

### 10.7 C++ GoCalExpressFee 稳定性

  

当前 C++ `GoCalExpressFee` 需要加强：

  

- URL 参数需要编码。

- JSON parse 失败要明确返回错误。

- `amount` 字段要兼容 number/string。

- Go 返回错误时要透传可读错误。

  

## 11. 推荐实施顺序

  

1. Go 侧完成包材主数据、规则表、区域表、规则包材明细表、唯一性校验。

2. Go 侧完成包材 + 快递费试算接口，返回快递费、包材费、包材重量、命中规则明细。

3. C++ `recipecache` 扩展重量计算，输出同一包材计费类型汇总后的 calc_list。

4. C++ `recipecache` 接入包材试算，并用药品重量 + 包材重量计算快递费。

5. `recipecache/calexpressfee` 返回 `express_fee + package_fee + package_detail`。

6. Deal 下单接入包材费收费项和校验。

7. 下单成功后写入包材最小计费记录，并输出包材计费详细日志。

8. 部分退款和整单退款接入包材费规则，并输出退款判断日志。

  

## 12. 最终建议

  

规则主表 + 区域关联表 + 包材明细表是合理设计，尤其在 MySQL 5.7 下比区域字符串、包材 JSON 或逗号字段更稳定。

  

唯一性校验必须按“交集冲突”实现，并配合 `GET_LOCK` 防并发重复保存。

  

C++ 侧只应负责已有药品重量和处方基础信息提取，不应重复实现 Go 包材规则。包材规则、包材费、包材重量、命中明细应由 Go 统一计算，保证下单、退款、对账口径一致。

  

## 13. 最终修订点

  

以下修订作为最终落地口径，覆盖前文中不够精确的表述：

  

1. 包材信息从规则主表拆出到 `t_package_fee_rule_material`。保存规则时，规则主表只保存匹配条件；包材明细保存 material、单价、重量、数量：

  

```go

detail.SfMaterialCode = material.SfMaterialCode

detail.MaterialName = material.MaterialName

detail.ShortName = material.ShortName

detail.WeightUnit = material.WeightUnit

  

if detail.PackageWeight == 0 {

    detail.PackageWeight = material.DefaultWeight

}

```

  

同一条规则可以配置多条包材明细，计算时按明细逐条累加金额和重量。

  

2. `area_code` 使用 `clinic_city_id`，表示门店/发货城市。收件城市 `city` 只用于快递费目的地匹配，不用于包材规则区域匹配。后台配置和下单计算必须使用同一套 `clinic_city_id`。

  

3. 金额字段建议统一类型。`express_fee` 和 `package_fee` 在 Go proto、C++、Deal 下单/退款链路中建议统一为 `int64`，避免一个有符号、一个无符号导致加减和序列化边界问题。

  

4. 管理端需要补规则配置查询接口，不能只依赖 auto_gen 的规则主表查询：

  

```proto

rpc QueryPackageFeeRuleConfig(PackageFeeRuleConfigFilter) returns (PackageFeeRuleConfigList) {}

rpc QueryPackageFeeRuleConfigDetail(PackageFeeRuleConfigFilter) returns (SavePackageFeeRuleConfigReq) {}

```

  

列表和详情都应返回规则主表 + `area_list`，方便后台回显和编辑。

  

5. 不再需要拆包阈值常量。类似 “大于 15 剂用 2 个包材” 的规则通过 `dose_min/dose_max + package_count` 配置完成：

  

```text

1-15 剂规则：rule_material.package_count = 1

16-9999 剂规则：rule_material.package_count = 2

```

### 15.1 包材规则配置：剂数匹配模式放宽（recipe_type=1/2/3）

```go
// recipe_type=4（膏方丸剂类）必须使用不限剂数模式
if rule.RecipeType == 4 && rule.DoseMatchMode != 2 {
```

### 15.2 包材规则冲突报错明细化

文件：同上，`checkRuleConflict` 及冲突报错信息（预检 + 事务内复检两处）。

`ruleConflictInfo` 从 `{RuleID, RuleName, AreaCode}` 扩展为：

```go
type ruleConflictInfo struct {
    RuleID             uint64
    RuleName           string
    AreaCode           string
    AreaName           string
    RecipeType         uint32
    DoseMatchMode      uint32
    DoseMin            int32
    DoseMax            int32
    EffectiveStartTime time.Time
    EffectiveEndTime   time.Time
}
```

新增 `formatConflictRule()`，把命中的冲突规则格式化为一句话，便于一眼定位：

```go
var recipeTypeNames = map[uint32]string{
    1: "代煎",
    2: "自煎",
    3: "代煎+自煎",
    4: "膏方丸剂类",
}

func formatConflictRule(c *ruleConflictInfo) string {
    doseDesc := "不限剂数"
    if c.DoseMatchMode == 1 {
        doseDesc = fmt.Sprintf("剂数区间%d-%d", c.DoseMin, c.DoseMax)
    }
    endDesc := c.EffectiveEndTime.Format("2006-01-02")
    if c.EffectiveEndTime.Unix() >= permanentEndUnix() {
        endDesc = "长期有效"
    }
    return fmt.Sprintf("规则「%s」（id=%d，%s，%s，生效时间%s~%s）",
        c.RuleName, c.RuleID, recipeTypeNames[c.RecipeType], doseDesc,
        c.EffectiveStartTime.Format("2006-01-02"), endDesc)
}
```

冲突查询 SQL 同步增加 `r.recipe_type/dose_match_mode/dose_min/dose_max/effective_start_time/effective_end_time`，以及通过 `COALESCE` 子查询取命中区域的 `area_name`（与 `area_code` 取法一致）。

#### 已知现象（非 bug）

`WorkStationRecipeAdd` 处理云端 `recipe_ext_obj` 为空的处方时，`AlterJsonExt` 会打印一条 `AlterJsonExt parse fail. old data: []` 的析构日志。`[%s]` 是日志格式里的字面方括号，`old data: []` 实际是 `old data: ""`（空字符串）。原因是 `strToJson("")` 解析失败，但 jsoncpp 对 `nullValue` 调用 `operator[]` 会自动提升为 `object`，所以 `AlterJsonExt` 仍能正确返回 `{"urgent_flag":"1",...}`，标识写入不受影响，该日志属于无害噪音。

### 15.6 已发货整单退不退包材费

#### 背景与目标

`cancelandwxrefund` 执行整单退款时，`GetRefundFee()` 以 `GetPrice()`（即 `ReceivablesPrice + FeeDiscountMoney - CouponDiscount`）作为退款基数。`GetPrice()` 包含包材费（前端 `recipe_total_money` 已含 `others_price`，下单时写入订单总价）。

目标：若订单已发货（`GetShipTime() > 0`），整单退不退包材费；未发货则全退。

#### 核心问题

`others_price`（即 `CDealExtentionDo.GetOtherPrice()`）存的是所有非医用耗材的合计金额，可能同时包含：

- 包材（塑料袋、冰袋等，`is_package_material: 1`）
- 非包材耗材（疫苗注射费、治疗耗材等）

无法直接把 `otherPrice` 整体扣除，需要单独存储包材费。

#### 方案

**1. 下单时区分并单独存储包材费**

前端在 `others_list` 数组的每个包材项目上新增字段 `is_package_material: "1"`（非包材项目不传或传 `"0"`）。

`ReadItemList()` 读取时，对 `is_package_material == "1"` 的项累加其 `total_price` 到新增输出参数 `ddwPackageFee`。

包材费单独存入 `CDealExtentionDo.ShipPackageFee`（新字段，下单路径写，退款路径读）。

同时为包材费在费用明细中单独出一个 fee type，便于后续对账区分。

**2. 退款时按发货状态扣除包材费**

`CancelAndWXRefund` 全单退路径中，在 `GetRefundFee()` 算出 `refundFee` 后、`SetRefundMoney()` 前插入：

```cpp
if (oDealDo.GetShipTime() > 0) {
    uint64_t ddwPkgFee = oDealDo.GetExtention().GetShipPackageFee();
    if (ddwPkgFee > 0 && refundFee >= ddwPkgFee) {
        refundFee -= ddwPkgFee;
    }
}
```
