# 处方合并发货 — 技术方案

> [!info] 说明
> 本文档为技术方案概览，供团队评审讨论。代码细节见执行方案。
> 需求文档：[[处方合并发货需求文档]]

---

## 一、总体思路

合并发货涉及 C++（deal_proj、recipe_proj）和 Go（express-service）两侧改动。

**调用链**：
```
前端 → C++ SplitRecipeAdd → CheckShipFee/GetShipFee
                          → calexpressfee CGI → Go ComputeShippingFee RPC
```

**合并维度**：订单（deal）维度，前端勾选的是订单，主副关系也是订单对订单。一个订单内的所有处方整体作为主订单或副订单参与合并，不允许部分参与。

**数据存储**：不新增数据库表，利用现有 `t_user2recipe_xxx` 表的 `strRecipeExt` JSON 字段扩展（存储实现在 recipe 上，概念上是订单维度）。

---

## 二、数据模型

不新增数据库表，仅扩展 deal_ext / recipe_ext JSON 字段。

> [!note] 存储层对应关系
> - **deal_ext** → `deal_user.t_deal_xxx`，`CDealExtentionDo` 对应的 ext JSON
> - **recipe_ext** → `db_recipe.t_user2recipe_xxx`（recipe_proj 的处方表），`queryrecipelist` 读的就是这里
> - 写入路径：deal_proj 在 `CRecipeDo.RecipeExt` 上设好值 → `CreateDeal` 写入 `deal_user.t_recipe_xxx` → 触发异步事件 `NormalRecipePayOK` / `ProcessPayOK` → `SendModifyRecipe` 同步到 `db_recipe.t_user2recipe_xxx`
> - 结论：我们在 WorkStationRecipeAdd 里写好 `CRecipeDo.RecipeExt`，剩余同步链路已有，无需额外处理

### deal_ext 新增字段（所有寄付订单下单时写入）

| 字段 key | 类型 | 含义 |
|---|---|---|
| `ship_merge_group_id` | string | 合并组 ID。默认写自身 deal_id；副订单合并时改写为主订单 deal_id。同组 ID = 同一合并组 |
| `is_ship_master` | int | 是否主订单。默认 1；副订单合并时改为 0 |
| `ship_master_deal_id` | uint64 | 主订单 deal_id（副订单才有实意；若用 ship_merge_group_id 推导可省略） |
| `ship_base_fee` | uint32 | 首重费（分），仅主订单有值，**退款时不退** |
| `ship_extra_fee` | uint32 | 本订单续重费总额（分），可选，续重各处方汇总 |
| `ship_unbind_orig_deal_id` | uint64 | **补收订单专用**：记录此补收单是为哪个副订单补收首重费 |

### recipe_ext 新增字段

| 字段 key | 类型 | 含义 |
|---|---|---|
| `ship_extra_fee` | int | 该处方分摊的续重费（分）。**退款时只退此金额**；首重费不退 |

> [!note] 核心设计原则
> - `ship_merge_group_id` 是合并组的纽带：查询某组所有订单 = 按 group_id 检索
> - 判断是否副订单：`ship_merge_group_id ≠ 自身 deal_id`
> - 判断是否主订单：`is_ship_master == 1`
> - 不存副订单列表（无需 `merge_ship_sub_deal_ids`），通过 `ship_merge_group_id == master_deal_id` 查整个合并组（含主订单），再加 `is_ship_master==0` 过滤得副订单
> - `ship_price`（已有字段）= 本次收的快递总费 = 主订单首重 + 续重总额 / 副订单仅续重

## 三、收费页面（合并候选列表）

需求：勾选"是否合并处方发货"后，展示可合并的主订单供选择。

**候选列表接口**：复用 `dealList`，过滤条件：
- 同 `clinic_id`（同门店）
- `ship_method = 寄付`
- 当日时间范围
- **未发运**：`posted_time == 0`
- **未被合并**（即仍是自己的主订单，未成为他人副订单）：`ship_merge_group_id == self_deal_id`（前端二次过滤，或 dealList 支持该过滤后端处理）
- 不限患者，不限地址，收费员自行判断

**选中订单后**：调 `getDeal(deal_id)` 获取完整处方详情，用于展示内容和预估续重费。

**合并前置校验**（下单时后端执行）：
1. 目标订单 A 已支付、寄付、同门店
2. A 未被合并（`ship_merge_group_id == A.deal_id`）
3. 代煎出库地点一致：比较 `GetStock` 扣库存的 `pharmacy_id` 是否相同

## 四、收费（下单）

### 4.1 运费计算规则

| 支付类型        | 运费处理                                                                                |
| ----------- | ----------------------------------------------------------------------------------- |
| 寄付（普通下单）    | calexpresfee接口返回 `ship_base_fee` + 各处方 `ship_extra_fee`，分别存储到 deal_ext / recipe_ext |
| 寄付（副订单合并下单） | 不收首重；calexpresfee接口返回各处方的续重费（副订单只付续重）                                               |
| 到付          | 各订单独立计费，无运费合并逻辑，物理合并发货                                                              |
| 包邮          | 无运费，物理合并发货                                                                          |

### 4.2 SplitRecipeAdd 新增参数

副订单下单时，前端传入：

```json
{
  "is_merge_ship": 1,
  "ship_merge_master_deal_id": 18363044
}
```

- `is_merge_ship=1` 标识合并下单；`ship_merge_master_deal_id` 为要合并进入的主订单 deal_id
- 后端从主订单继承：`ship_method`、`receiver_name`、`receiver_phone`、`address`、`province`、`city`、`area`
- 后端看到 `is_merge_ship=1` 即传 `calc_mode=1` 给 calexpressfee，不收首重

### 4.3 calexpressfee / Go 接口变化

**入参**（含各处方重量）：

- 普通下单：传本订单处方列表（`recipe_list` 含各处方重量）
- 合并下单：传**主订单处方 + 副订单处方**合并列表（Go 需全量重量正确分配续重），携带 `calc_mode=1`

**出参**（per-recipe 续重费明细）：

```json
{
  "ship_base_fee": 1200,
  "recipe_fees": [
    {"recipe_id": 101, "second_weight_fee": 200},
    {"recipe_id": 102, "second_weight_fee": 0}
  ]
}
```

- `calc_mode=1` 时 `ship_base_fee` 返回 0（不收首重）
- `recipe_fees` 只含本次副订单的处方（主订单处方已结算，Go 只返回新处方的续重）

`GetShipFee` 需新增输出结构携带 per-recipe 明细，下单后写回 ext 用。

### 4.4 下单成功后写回 Ext

**所有寄付订单**下单后，deal_ext 写入（含非合并订单）：

- `ship_merge_group_id` = 自身 deal_id（默认值；副订单下单后改写为主订单 deal_id）
- `is_ship_master` = 1（默认；副订单改为 0）
- `ship_base_fee` = Go 返回的首重费（副订单为 0）
- `ship_extra_fee` = 本订单续重费总额（各处方汇总）

**所有寄付订单**每个 recipe 写入 recipe_ext：

- `ship_extra_fee` = Go 返回的该处方续重费

**副订单额外**更新 deal_ext：

- `ship_merge_group_id` → 主订单 deal_id
- `is_ship_master` → 0
- `ship_master_deal_id` → 主订单 deal_id

## 五、退费（仍需商榷）

### 5.1 退款基本规则

- **首重费（`ship_base_fee`）非整单退款时不退**：记在主订单 deal_ext
- **续重费**：按处方退，退 recipe_ext.`ship_extra_fee`（各处方独立）

### 5.2 副订单退费

直接退该副订单的药品费 + 各处方续重费（读 `ship_extra_fee`）。主订单及其他副订单不受影响。

### 5.3 主订单退费 — 已发运

`posted_time > 0`，不影响副订单状态，退药品费（快递已发，不退首重）。

### 5.4 主订单退费 — 未发运

`posted_time == 0` 时退主订单，副订单失去主单：

- 退主订单药品费（全部快递费）
- 副订单需补收首重费才能独立发货：走解绑流程（生成补收订单）

> [!question] 待确认
> 主订单未发运退费时，系统是强制要求先解绑副订单，还是退费后仅提示补收？退费和补收的顺序需产品确认。

## 六、解绑

**接口**：C++ 新增专用接口 `/cgi-bin/deal/unbind_ship_merge`，一次完成解绑 + 建补收单全流程。

**权限**：仅副订单可发起；已发运、已审方不能解绑。

### 6.1 寄付副订单解绑流程（新接口内部顺序）

1. **校验**：`is_ship_master == 0`（是副订单）、`posted_time == 0`（未发运）、未审方
2. **清除副订单 deal_ext 合并字段**：`ship_merge_group_id` 重置为自身 deal_id，`is_ship_master=1`，`ship_master_deal_id=0`，`ship_extra_fee=0`
3. **清除副订单各处方 recipe_ext.`ship_extra_fee`**
4. **重算独立首重费**：对该副订单自身处方调 `GetShipFee`（普通首重模式，`iCalcMode=0`）
5. **创建补收订单**：`DEAL_TYPE_OTHERS` + `FEE_TYPE_SHIP`，金额 = 步骤 4 的首重费，deal_ext 写 `ship_unbind_orig_deal_id` = 副订单 deal_id
6. **返回**：补收单 deal_id，前端凭此走支付流程
7. 主订单数据不修改

> [!note] 为什么用专用接口而不复用 Others
> 解绑是"校验 → 清 ext → 算费 → 建单"的完整原子流程，放在一个接口里一致性更好；`Others` 接口参数过于通用，缺少合并发货的上下文（clinic_id、orig_deal_id 等）。

### 6.2 到付副订单解绑

仅执行步骤 1-3（校验 + 清 ext），无需建补收单，各自独立发货。

## 七、发货控制

### 7.1 拦截锁死（待补收）

- 检查 `merge_ship_fee_state == 2`，禁止点击发货
- 提示："运费待补收，请先处理费用"

### 7.2 发货提醒弹窗

- 无论寄付/到付，点击发货时均弹窗提示合并编号
- 弹窗内容："合并发货处方，合并处方编号 XXX/XXX，请核对合并发货"
- 需 Go 新增接口：给定 recipe_id，返回合并组所有处方编号

### 7.3 快递单号与打印

- 合并组统一生成 **1 个快递单号**（`mail_no`）
- 快递单号在**审方时**由 `CreateExpressBillProxy` 向快递公司下单生成，不是发货时
- 快递单号存储在处方侧（`db_recipe.t_user2recipe_xxx.Fdeliver_no`），订单侧不单独存储
- **共享机制**：副订单下单成功后，C++ 调 Go API `POST /recipe/handler/InsertUnirecipeGroup`，把主副订单所有处方的 `recipe_id` 一起加入同一个 `UnirecipeGroup`。审方时第一个处方调 `CreateExpressBillProxy` 生成 `mail_no` 并写回组内所有处方，后续处方直接复用，无需修改 `CreateExpressBillProxy` 逻辑
- 主订单打印运单号，副订单**不打印运单号**

> [!note] 合并发药 vs 合并发货
> 系统已有"合并发药"功能（`InsertUnirecipeGroup`），也是处方分组共享 `mail_no` 的机制。合并发货复用该底层机制，两者区别在于：合并发药是药房人工操作、无费用变化；合并发货是收费下单时确定、有首重/续重分摊。

## 九、Go 侧改动

### 9.1 ComputeShippingFee（现有 RPC，扩展参数）

新增 `calc_mode` 字段：
- `calc_mode=0`：现有逻辑（首重）
- `calc_mode=1`：续重计算
  - 内部配置路径：`sub_billing_weight_kg × ContinuedWeightPrice`
  - SF 实时询价路径：`SF.price(firstWeight + sub_billing_weight) - SF.price(firstWeight)`（两次调用取差）
  - `sub_billing_weight` = 副订单药品重 + 副订单包材重（包材重由 `package_calc_list` 计算，已有逻辑）
  - `firstWeight` 用 `shippingConfig.FirstWeight`（配置值，勿硬编码）

### 9.2 新增接口：发货前弹窗查询

给定 recipe_id，返回合并组所有订单的处方编号信息。

### 9.3 新增接口：解绑副订单

清除合并关联 ext 字段（或触发补收流程）。
