# 复诊提醒弹窗逻辑优化需求

## 1. 背景

**当前逻辑**：前端通过 `queryOptionForWorkStation` 返回的 `followupReminder` 字段决定是否调用 `CheckFollowUp`，进而决定是否弹出预约复诊弹窗。

**新逻辑**：不再依赖 `followupReminder`，改为始终调用 `CheckFollowUp`，根据返回的 `has_follow_up`、新增的 `follow_up_type` 以及患者就诊类型（首诊/续诊）统一控制弹窗行为。同时，诊结接口 `diagnose/end` 新增 `follow_up_type` 入参，记录医生选择的复诊建议类型。

---

## 2. 接口变更

### 2.1 CheckFollowUp

**路径**：`deal/handler/CheckFollowUp`

**请求参数**：

| 参数名       | 类型     | 必填  | 说明         |
| --------- | ------ | :-: | ---------- |
| `deal_id` | uint64 |  是  | 挂号单 ID（新增） |


**响应参数**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `has_follow_up` | int32 | 0 = 无预约复诊；1 = 已预约复诊 |
| `follow_up_type` | int32 | 新增。0 = 未设置；1 = 无需复诊；2 = 不适随诊；3 = 事后随诊 |

---

### 2.2 诊结接口 diagnose/end

**路径**：`/cgi-bin/diagnose/end`

**新增入参**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| `follow_up_type` | string | 是 | 医生选择的复诊建议类型，取值 `"1"`/`"2"`/`"3"` |


---

## 3. 业务逻辑

### 3.1 调用规则

1. 不再读取 `queryOptionForWorkStation` 的 `followupReminder` 字段。
2. 医生站和收费站均需调用 `CheckFollowUp`，并传入 `deal_id`。
3. 医生站和收费站的弹窗判断逻辑不同，分别见 3.2 和 3.3。

### 3.2 医生站弹窗触发规则

医生站在诊结时触发，根据就诊类型（首诊/续诊）判断。

| 就诊类型 | `has_follow_up` | `follow_up_type` | 是否弹窗 |
| :--: | :-------------: | :--------------: | :--: |
|  首诊  |        0        |    0（首诊时未设置）     |  弹窗  |
|  首诊  |        1        |        任意        | 不弹窗  |
|  续诊  |        1        |        任意        | 不弹窗  |
|  续诊  |        0        |    1 或 2 或 3     | 不弹窗  |
|  续诊  |        0        |        0         |  弹窗  |

> - 首诊时尚未诊结，`follow_up_type` 必然为 0，因此首诊只需判断 `has_follow_up`。
> - 续诊时，只要医生之前已选过任意复诊类型（1/2/3），无论选的是哪项，均不再弹窗。

### 3.3 收费站弹窗触发规则

收费站仅根据 `follow_up_type` 和 `has_follow_up` 判断是否放行。

| `has_follow_up` | `follow_up_type` | 是否弹窗 |
| :-------------: | :--------------: | :--: |
|        1        |        任意        | 不弹窗  |
|       任意        |     1（无需复诊）      | 不弹窗  |
|        0        |     0、2 或 3      |  弹窗  |

> 只有"已预约复诊"或医生明确选择"无需复诊"两种情况才直接放行；选了"不适随诊"或"事后随诊"但无预约时，仍需弹窗提示。

### 3.4 弹窗交互要求

弹窗提供三个必选项，用户选择后方可完成诊结：

| 选项文案 | `follow_up_type` 值 |
|----------|:-------------------:|
| 无需复诊 | 1 |
| 不适随诊 | 2 |
| 事后随诊 | 3 |

选中后在调用 `diagnose/end` 时将所选值传入 `follow_up_type`。

---

## 4. 测试要点

**医生站**

1. 首诊 + `has_follow_up=0`：必须弹窗，三个选项均可选，提交后 `end` 收到正确的 `follow_up_type`。
2. 首诊 + `has_follow_up=1`：不弹窗，正常结束诊结。
3. 续诊 + `has_follow_up=0` + `follow_up_type=0`：弹窗（未选过）。
4. 续诊 + `has_follow_up=0` + `follow_up_type=1/2/3`：不弹窗（已选过任意类型）。
5. 续诊 + `has_follow_up=1`：不弹窗。

**收费站**

6. `has_follow_up=1`：不弹窗（已预约复诊）。
7. `follow_up_type=1` + `has_follow_up=0`：不弹窗（无需复诊）。
8. `follow_up_type=2/3` + `has_follow_up=0`：弹窗。
9. `follow_up_type=0` + `has_follow_up=0`：弹窗（未选过）。
