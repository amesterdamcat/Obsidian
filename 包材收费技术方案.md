# **1. 背景与目标**

寄付下单当前通过顺丰收费标准接口计算快递费。系统将药品重量和收件地址传给顺丰，顺丰返回快递费，由患者支付。本次补齐包材处理能力：包材费由我方系统独立收取，包材重量并入药品重量后传给顺丰作为计费重量。

· 后台支持维护包材收费规则，并支持一条规则适用于多个区域。

· 下单时根据区域、处方类型、剂数或特殊预设匹配包材规则。

· 下单成功时保存包材计费快照，后续配置变更不追溯历史订单。

· 退费、对账、纠纷处理均以订单快照为准。

# **2. 业务规则**

## **2.1 费用计算规则**

计费重量 = 药品重量 + 包材总重量  
包材总重量 = 单个包材重量 × 包材数量  
包材费 = 包材单价 × 包材数量  

· 15 剂以上系统自动拆 2 个包装，包材数量 = 2。

· 15 剂及以下包材数量 = 1。

· 包材费不进入顺丰计费，只作为我方独立费用项向患者收取。

· 包材重量参与顺丰计费重量。

## **2.2 费用项展示**

|   |   |   |   |
|---|---|---|---|
|**费用项**|**来源**|**入账科目**|**说明**|
|快递费|顺丰收费标准接口返回|快递代收|按药品重量 + 包材重量计算|
|包材费|我方包材规则计算|包材收入|单价 × 包材数量|

## **2.3 匹配维度**

|   |   |
|---|---|
|**维度**|**说明**|
|区域|下单区域必须命中规则适用区域。|
|处方类型|1 代煎，2 自煎，3 代煎+自煎。类型 3 同时覆盖代煎和自煎。|
|剂数|普通剂数按区间匹配，例如 1-7、8-9999、自定义 7-10。|
|特殊预设|膏方、丸剂等不按剂数区间匹配，按特殊预设编码匹配。|
|生效时间|当前时间需要落在 effective_start_time 和 effective_end_time 之间。|

|   |   |
|---|---|
|**模块**|**职责**|
|包材主数据模块|维护系统内置包材库，业务侧只能选择，不能新建。|

# **5. 剂数预设与匹配设计**

|   |   |   |   |   |   |
|---|---|---|---|---|---|
|**页面选择**|**dose_match_mode**|**dose_preset_code**|**dose_min**|**dose_max**|**匹配方式**|
|7剂以内|1 剂数区间|LE_7|1|7|dose_min <= 剂数 <= dose_max|
|7剂以上|1 剂数区间|GT_7|8|9999|dose_min <= 剂数 <= dose_max|
|自定义 7-10剂|1 剂数区间|NULL|7|10|dose_min <= 剂数 <= dose_max|
|膏方|2 特殊预设|GAOFANG|NULL|NULL|dose_preset_code = GAOFANG|
|丸剂|2 特殊预设|WANJI|NULL|NULL|dose_preset_code = WANJI|

说明：LE_7、GT_7 不建议在下单时直接用 dose_preset_code 匹配。普通处方下单通常只有剂数，因此保存规则时需要将普通预设转换成剂数区间。GAOFANG、WANJI 属于特殊业务类型，不按剂数区间匹配。

# **6. 生效时间设计**

|   |   |   |
|---|---|---|
|**页面选择**|**落库字段**|**落库值**|
|立即生效|effective_start_type|1|
|立即生效|effective_start_time|当前日期 00:00:00|
|指定时间生效|effective_start_type|2|
|指定时间生效|effective_start_time|用户选择的时间|
|长期有效|effective_end_type|1|
|长期有效|effective_end_time|9999-12-31 23:59:59|
|指定结束时间|effective_end_type|2|
|指定结束时间|effective_end_time|用户选择的结束时间|

· 下单匹配和冲突判定只看 effective_start_time、effective_end_time。

· effective_start_type、effective_end_type 主要用于前端回显和操作日志。

· 长期有效虽然数据库保存为 9999-12-31 23:59:59，但页面展示为“长期有效”。

# **7. 下单匹配 SQL**

## **7.1 普通剂数区间匹配**

SELECT r.*  
FROM t_package_fee_rule r  
JOIN t_package_fee_rule_area a ON r.id = a.rule_id  
WHERE a.area_code = ?  
  AND r.status = 1  
  AND r.effective_start_time <= NOW()  
  AND r.effective_end_time >= NOW()  
  AND r.recipe_type IN (?, 3)  
  AND r.dose_match_mode = 1  
  AND r.dose_min <= ?  
  AND r.dose_max >= ?  
ORDER BY  
  CASE WHEN r.recipe_type = ? THEN 0 ELSE 1 END,  
  r.dose_min DESC,  
  r.id ASC  
LIMIT 1;  

## **7.2 特殊预设匹配**

SELECT r.*  
FROM t_package_fee_rule r  
JOIN t_package_fee_rule_area a ON r.id = a.rule_id  
WHERE a.area_code = ?  
  AND r.status = 1  
  AND r.effective_start_time <= NOW()  
  AND r.effective_end_time >= NOW()  
  AND r.recipe_type IN (?, 3)  
  AND r.dose_match_mode = 2  
  AND r.dose_preset_code = ?  
ORDER BY  
  CASE WHEN r.recipe_type = ? THEN 0 ELSE 1 END,  
  r.id ASC  
LIMIT 1;  

# **8. 唯一插入冲突判定**

唯一性不能只依赖数据库唯一索引，因为涉及多区域、类型覆盖、剂数区间交集和生效时间交集。保存前需要进行业务冲突查询。

|   |   |
|---|---|
|**冲突维度**|**判定规则**|
|区域|新规则区域集合与已有规则区域存在交集。|
|处方类型|代煎与代煎冲突，代煎与代煎+自煎冲突；自煎同理；代煎+自煎与所有类型冲突。|
|普通剂数区间|已有 dose_min <= 新 dose_max 且已有 dose_max >= 新 dose_min。|
|特殊预设|dose_preset_code 相同。|
|生效时间|已有开始 <= 新结束 且已有结束 >= 新开始。|

## **8.1 处方类型冲突集合**

|   |   |
|---|---|
|**新规则 recipe_type**|**会冲突的已有 recipe_type**|
|1 代煎|1 代煎、3 代煎+自煎|
|2 自煎|2 自煎、3 代煎+自煎|
|3 代煎+自煎|1 代煎、2 自煎、3 代煎+自煎|

## **8.2 普通剂数区间冲突 SQL**

SELECT  
    r.id,  
    r.rule_name,  
    a.area_code,  
    a.area_name,  
    r.recipe_type,  
    r.dose_match_mode,  
    r.dose_preset_code,  
    r.dose_min,  
    r.dose_max,  
    r.effective_start_time,  
    r.effective_end_time  
FROM t_package_fee_rule r  
JOIN t_package_fee_rule_area a ON r.id = a.rule_id  
WHERE r.status = 1  
  AND a.area_code IN (?, ?, ?)  
  AND r.id <> ?  
  AND r.recipe_type IN (?, ?, ?)  
  AND r.dose_match_mode = 1  
  AND r.dose_min <= ?  
  AND r.dose_max >= ?  
  AND r.effective_start_time <= ?  
  AND r.effective_end_time >= ?  
LIMIT 1;

## **8.4 并发控制**

-- 保存前获取锁  
SELECT GET_LOCK('package_fee_rule_save_lock', 10);  
  
-- 保存完成后释放锁  
SELECT RELEASE_LOCK('package_fee_rule_save_lock');  

· 后台配置并发不高，第一版建议使用全局锁，简单可靠。

· 加锁后在事务内完成：冲突查询、插入主表、插入区域表、写操作日志。

# **10. 下单流程改造**

|   |   |
|---|---|
|**步骤**|**处理内容**|
|1|接收下单请求，识别区域、处方类型、剂数、特殊预设、药品重量。|
|2|按区域 + 类型 + 剂数/特殊预设 + 生效时间匹配包材规则。|
|3|如果未命中规则，拦截下单并提示联系管理员。|
|4|如果命中规则但包材已停用，拦截下单并推送管理员告警。|
|5|计算包材数量、包材费、包材总重量、计费重量。|
|6|使用计费重量调用顺丰收费标准接口。|
|7|生成订单费用明细：快递费、包材费。|
|8|事务内创建订单、费用明细、包材计费快照、支付单。|

package_count = dose_count > 15 ? 2 : 1  
package_fee_amount = unit_price * package_count  
total_package_weight = package_weight * package_count  
billing_weight = drug_weight + total_package_weight  

# **11. 退费规则**

|   |   |
|---|---|
|**场景**|**包材费处理**|
|未发货整单退|全退|
|未发货部分退，且不影响剂数档位|不退，按整单计|
|已发货退费|不退|
|合并寄付场景|包材费按处方独立计、独立退，不参与首重续重分摊|

# **12. Go 1.14 实现要点**

Go 1.14 实现时建议使用 database/sql，不使用泛型。保存规则入口建议按以下顺序组织。

func (s *PackageFeeRuleService) CreateRule(req CreatePackageFeeRuleReq) error {  
    if err := validateCreateReq(req); err != nil {  
        return err  
    }  
  
    doseInfo, err := ResolveDoseMatchInfo(req)  
    if err != nil {  
        return err  
    }  
  
    timeInfo, err := ResolveRuleTimeInfo(req)  
    if err != nil {  
        return err  
    }  
  
    conflictRecipeTypes, err := GetRecipeTypeConflictSet(req.RecipeType)  
    if err != nil {  
        return err  
    }  
  
    locked, err := s.getLock("package_fee_rule_save_lock", 10)  
    if err != nil {  
        return err  
    }  
    if !locked {  
        return errors.New("当前有其他人正在保存包材规则，请稍后再试")  
    }  
    defer s.releaseLock("package_fee_rule_save_lock")  
  
    tx, err := s.DB.Begin()  
    if err != nil {  
        return err  
    }  
    defer func() {  
        if err != nil {  
            _ = tx.Rollback()  
        }  
    }()  
  
    conflict, err := s.findConflictRule(tx, req.AreaList, 0, conflictRecipeTypes, doseInfo, timeInfo)  
    if err != nil {  
        return err  
    }  
    if conflict != nil {  
        _ = tx.Rollback()  
        return fmt.Errorf("当前配置与已有包材规则冲突：区域=%s，规则=%s", conflict.AreaName, conflict.RuleName)  
    }  
  
    ruleID, err := s.insertRule(tx, req, doseInfo, timeInfo)  
    if err != nil {  
        return err  
    }  
  
    if err = s.insertRuleAreas(tx, ruleID, req.AreaList); err != nil {  
        return err  
    }  
  
    return tx.Commit()  
}  

## **12.1 剂数转换**

func ResolveDoseMatchInfo(req CreatePackageFeeRuleReq) (DoseMatchInfo, error) {  
    if req.IsCustomDose {  
        if req.DoseMin <= 0 || req.DoseMax <= 0 || req.DoseMin > req.DoseMax {  
            return DoseMatchInfo{}, errors.New("自定义剂数区间不合法")  
        }  
        return DoseMatchInfo{DoseMatchMode: 1, DoseMin: req.DoseMin, DoseMax: req.DoseMax}, nil  
    }  
  
    switch req.DosePresetCode {  
    case "LE_7":  
        return DoseMatchInfo{DoseMatchMode: 1, DosePresetCode: "LE_7", DoseMin: 1, DoseMax: 7}, nil  
    case "GT_7":  
        return DoseMatchInfo{DoseMatchMode: 1, DosePresetCode: "GT_7", DoseMin: 8, DoseMax: 9999}, nil  
    case "GAOFANG":  
        return DoseMatchInfo{DoseMatchMode: 2, DosePresetCode: "GAOFANG"}, nil  
    case "WANJI":  
        return DoseMatchInfo{DoseMatchMode: 2, DosePresetCode: "WANJI"}, nil  
    default:  
        return DoseMatchInfo{}, fmt.Errorf("不支持的剂数预设编码：%s", req.DosePresetCode)  
    }  
}

# **15. 最终结论**

· 区域适用关系采用拆表，不使用 JSON 数组。

· 长期有效使用 9999-12-31 23:59:59，避免 NULL 带来的 SQL 复杂度。

· LE_7、GT_7 保存时转换为剂数区间；GAOFANG、WANJI 作为特殊预设匹配。

· 下单匹配只看实际生效时间字段，不依赖生效类型字段。

· 唯一冲突判定通过业务 SQL 完成，并结合 MySQL GET_LOCK 防并发重复保存。

· 订单成功后必须保存包材计费快照，历史订单不受后续配置变更影响。
