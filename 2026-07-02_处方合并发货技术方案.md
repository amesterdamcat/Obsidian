# 处方合并发货技术方案

**Goal:** 多个处方订单合并发货，主订单承担首重，副订单分摊续重，统一1个快递单号（快递单号是药房发药才产生）
**方案:** 不新增数据库表，仅扩展 deal_ext / recipe_ext JSON字段

***

## 一、术语

| 术语          | 定义                               |
| ----------- | -------------------------------- |
| 处方 (Recipe) | 独立医疗单元，`cloud_recipe_id`         |
| 处方订单 (Deal) | 1+个处方组成的收费订单，`deal_id`，快递费在此级别计算 |

***

## 二、现状

快递费在订单级别计算：`deal.cpp` → `GetShipFee()` → `calexpressfee` → `Fship_price`

问题：每个订单各付首重费、各自发货。期望合并为一个包裹。

***

## 三、技术方案

### 3.1 合并组概念

* 同组订单 deal_ext.`ship_merge_group_id` 值相同

* 主订单（最早那个）承担首重，副订单分摊续重

* 副订单继承主订单的快递信息（收件人、地址等）

* 不限患者，不限地址，收费员自行判断

### 3.2 数据模型（不新增表）

**deal_ext 新增字段**：

| 字段                     | 类型     | 说明                                                    |
| ---------------------- | ------ | ----------------------------------------------------- |
| ship_merge_group_id    | string | 合并组ID，同组相同，下单的时候默认是自己的deal_id，合并的时候把主订单号写入到副订单的该合并组ID |
| is_ship_master         | int    | 是否主订单 0/1，每次下单都写入，默认是1，自己是自己的主订单                      |
| ship_master_deal_id    | uint64 | 主订单ID，如果合并组id使用订单id，那这个就不需要了，直接使用合并组id判断就知道主订单id是什么了  |
| ship_base_fee  | uint32 | 首重费(分)，仅主订单，退款不退                                      |
| ship_extra_fee | uint32 | 续重费(分)，当前是续重费各个处方自己保存自己的，订单要不要可以后面考虑                  |
| ship_express_no        | string | 快递单号，仅主订单，合并组共用（当前是不是已经有快递单号了？药房发货后才有）                |
|                        |        |                                                       |

**recipe_ext 新增字段**：

| 字段             | 类型  | 说明                  |
| -------------- | --- | ------------------- |
| ship_extra_fee | int | 该处方分摊续重费(分)，退款时退此金额 |

### 3.3 接口

#### 3.3.1 查询可合并订单 /cgi-bin/deal/dealList
可以过滤当日未合并(ship_merge_group_id!=deal_Id的)未发运（posted_time!=0）
getDeal或者queryrecipelist获取处方信息

#### 3.3.2 下单时合并（扩展现有下单接口SplitRecipeAdd）

新增参数：`is_merge_ship`（1代表合并）
从目标订单A继承：ship_method, receiver_name/phone/address/province/city/area等等所需数据

业务逻辑：

1. 创建订单B

2. 校验A：已支付、寄付、同门店、未加入合并组（合并组id仍是自己订单号）

3. 从A继承快递信息更新B

4. 校验代煎出库地点一致    去看 GetStock 扣库存的pharmacy_id 一致就是同一个出库地点

5. checkshipfee校验这部分得思考一下要怎么校验，和前端下单前计算得到B订单各个处方的续重费一样的逻辑

6. 更新B deal_ext：ship_merge_group_id(同A), is_ship_master=0, ship_master_deal_id=A, Fship_price=（各个处方续重费）

7. 更新B订单各处方 recipe_ext.ship_extra_fee

#### 3.3.3 解绑 /cgi-bin/deal/unbind_ship_merge

* 仅副订单可解绑，已审方、已发运不能解绑

* 新增专用接口，一次完成完整流程（不复用 Others 接口）：
  1. 校验副订单合法性
  2. 清除B的deal_ext合并字段（ship_merge_group_id重置为自身deal_id，is_ship_master=1等）
  3. 清除B各处方 recipe_ext.ship_extra_fee
  4. 重算B独立首重费（GetShipFee 普通模式）
  5. 内部创建补收订单（DEAL_TYPE_OTHERS + FEE_TYPE_SHIP），deal_ext写 ship_unbind_orig_deal_id=B的deal_id
  6. 返回补收单 deal_id，前端走支付

* 合并组只剩主订单，不影响，主订单数据不修改

#### 3.3.4 药房发货

* 主订单生成快递单号写入 deal_ext.ship_express_no

* 副订单复用主订单的 ship_express_no

#### 3.3.5 主/副订单部分退款

* 快递费只退当前处方的续重部分

* 首重费不退（记在主订单deal_ext）

* 清除该处方的 ship_extra_fee（需要否？感觉不需要）
